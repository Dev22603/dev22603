## 3) Hyperlab Sportech Pvt Ltd

I worked at Hyperlab Sportech Pvt Ltd as a Software Engineer from September 22, 2025 to December 31, 2025. This was a full-time role, and it was one of the most technically different experiences I had because it involved Lua, hardware-connected systems, BLE, RFID, sensor data, and a web platform. 

Hyperlab was an early-stage startup, and I joined because I could see strong growth potential there. 

### Main product context

The core product I worked on was an exercise mat. The mat included:
- 2 LiDAR sensors
- 2 force sensors
- an LED screen with 320 x 240 resolution
- an EC11 rotary encoder
- BLE connectivity 

This setup was used to detect and track physical exercises. My main responsibility was to design and implement exercise detection algorithms using sensor readings from the four sensors. 

### Learning curve and algorithm work

When I joined, I had no previous experience with Lua. I also had no previous experience in hardware programming. For the first two days, I received knowledge transfer on the exercise mat, the company’s IDE, internal setup, and Claude Code. 

My early phase was difficult. For around the first month, I struggled because both the language and the hardware domain were new to me, and the exercise detection logic was not accurate enough in the beginning. The mat could sometimes be fooled into thinking an exercise had been done properly when it had not. 

Over time, I improved the algorithms significantly. By the end of the first month, I was creating much stricter and more accurate logic, and the exercise mat became far harder to fool. In total, I built algorithms for around 35 exercises. 

One measurable result from this period was improving the detection speed of the Mountain-Climb exercise by 55 percent. 

### How I used AI in this role

At Hyperlab, I used Claude Code heavily, along with Cursor, and Windsurf, but I used them as productivity tools, not as a replacement for technical thinking. Roughly 90 percent of the code was AI-generated with about 10 percent manual changes, which gave a large productivity boost while I kept full ownership of the logic and decisions. I created detailed context documents about the exercise mat, the sensors, and the exercise logic, and then used those documents to get cleaner and faster code generation. 

The algorithms, the logic decisions, the strictness rules, and the thinking about how the mat should detect real versus fake movement were driven by me. Claude Code helped me turn that logic into clean implementation faster, while I still owned the problem solving, validation, and decision making. 

This is important to my working style: I do not treat AI as something that thinks for me. I use it to speed up execution when I already understand what needs to be built. 

### Competition platform and Burnhouse

Later in my Hyperlab time, I was given a new task to build a full stack platform using Next.js and Supabase. This platform was built for a competition called Burnhouse. 

The idea was that multiple people would open the website on their phones and connect their exercise mats through BLE. Once all mats were connected, an admin could start the competition. When the competition started, devices would communicate with the mats, the Lua exercise logic would begin running, and the system would track performance in real time. 

The competition had eight exercises grouped into waves. A wave could include examples like 12 push-ups, 30 sit-ups, 20 spot running, and 30 mountain climbers. Each wave had a time limit, and the system had to determine how many waves a participant could complete before failing. 

For every repetition, the exercise mat would send a BLE event, the phone would trigger an API call, and the backend would update the competition state. I built a platform that processed these BLE event streams and supported real-time exercise tracking and leaderboards. Real-time updates were handled using Supabase real-time subscriptions, which meant no custom WebSocket server was required. The live leaderboard showed each participant's rep count, current wave, and whether they were still active or eliminated. 

### RFID and Lua UI flow

I also built a Lua-based UI flow for multi-user exercise sessions. This flow involved RFID-based interactions as well as app-based events, retries, and time-based rules. 

One version of the flow worked like this:
- A user scanned an RFID card.
- The user moved to the exercise mat.
- The user scanned the exercise using the RFID card on the mat.
- The mat guided the user through all assigned exercises.
- After completion, the user was redirected in the flow without needing to scan the RFID card again. 

### LiDAR dashboard

I also created a React dashboard for engineers to upload LiDAR CSV logs, visualize sensor readings on interactive Recharts graphs, annotate anomalies at exact timestamps, zoom, pan, and explore the data freely, and export filtered datasets. This helped reduce debugging effort and made sensor analysis more practical. 

### Coding practices

Despite the startup pace and a new language, I kept my Lua code clean, structured, and readable, with proper naming, logical organization, comments for complex logic, and modular design for reusability.

### Key takeaways from Hyperlab

This role proved that I can learn a very unfamiliar domain fast and still become productive. It also showed that I can handle a difficult mix of hardware context, algorithm design, real-time events, Lua development, and web platform work in the same role.
